# 阻塞清除

## 概述

当 Task 被阻塞时，Scrum Master Agent 负责识别阻塞原因、协调资源解决问题、更新状态文件恢复开发流畅。阻塞可能来自依赖未满足、外部依赖（API/服务）、需求不明确、技术难题等多种原因。

**在流程中的位置：** 进度监控发现阻塞 → **阻塞清除** → 恢复正常执行

## 触发条件

- 进度监控发现 Task 状态为 `blocked`
- Agent 主动上报遇到阻塞
- Task 超过预期时间 2 倍仍未完成
- 依赖链断裂（上游 Task 未完成影响下游）

## 输入

- `sprint/sprint-status.md`（查看阻塞项列表）
- 被阻塞 Task 的详细信息（`backlog/tasks/T-XXX.md`）
- Agent 的工作记录（了解阻塞详情）
- 项目环境信息（依赖、配置、外部服务状态）

## 流程

1. **识别阻塞**：从 `sprint-status.md` 的阻塞项列表中获取当前阻塞
2. **分析原因**：
   - **依赖阻塞**：上游 Task 未完成 → 催办上游 Agent 或调整优先级
   - **需求不明确**：验收标准模糊 → 联系 PO Agent 澄清
   - **技术难题**：方案不可行或有 bug → 协调 Dev Agent 排查
   - **外部依赖**：第三方 API 不可用 → 寻找替代方案或等待
   - **环境问题**：构建/测试环境异常 → 修复环境
3. **评估影响范围**：判断阻塞是否影响其他 Task（依赖链传播）
4. **制定解决方案**：
   - 能立即解决的 → 协调资源立即处理
   - 需要等待的 → 评估替代方案
   - 无法解决的 → 调整里程碑范围，移除或延后该 Task
5. **执行解决**：协调相关 Agent 执行解决方案
6. **更新状态**：
   - 解决后，将 Task 状态从 `blocked` 改回 `in-progress`
   - 更新 `sprint-status.md` 阻塞项列表
   - 记录阻塞原因和解决方案（避免重复发生）
7. **通知相关方**：告知受影响的 Agent 阻塞已解除

## 输出

- 更新后的 `sprint/sprint-status.md`（阻塞项更新）
- 阻塞处理记录（存入 `sprint/blocker-log.md`）
- 通知相关 Agent

## 文件规范

### 阻塞日志格式

路径：`sprint/blocker-log.md`

```markdown
# 阻塞日志

## BLK-001
- **Task:** T-006 支付集成
- **发现时间:** 2026-03-29 14:00
- **阻塞原因:** 第三方支付 API 文档缺失
- **影响范围:** T-007 订单确认（依赖 T-006）
- **解决方案:** 使用模拟 API 先完成开发，待文档就绪后替换
- **解除时间:** 2026-03-29 15:30
- **处理人:** Scrum Master
```

## 注意事项

- **快速响应**：阻塞是效率的最大敌人，发现后应立即处理
- **记录所有阻塞**：即使快速解决的也要记录，积累经验用于预防
- **区分阻塞和正常延迟**：Task 慢不等于阻塞，只有无法继续推进才是阻塞
- **不要替 Agent 做技术决策**：技术方案的调整应协调 Dev Agent 而非直接决定
- **关注级联影响**：一个阻塞可能引发多个下游阻塞，需要全面评估
