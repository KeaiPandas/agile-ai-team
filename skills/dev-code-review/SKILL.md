# Code Review Skill

审查代码质量，提供改进建议。

## 工作流程

### 1. Review 准备

**获取代码：**
```bash
# 查看变更文件
git diff --name-only

# 查看具体变更
git diff
```

**确认范围：**
- [ ] 理解变更背景
- [ ] 确认相关需求
- [ ] 了解技术上下文

### 2. Review 维度

**功能正确性：**
- [ ] 是否实现了需求
- [ ] 边界条件是否处理
- [ ] 错误情况是否覆盖

**代码质量：**
- [ ] 命名是否清晰
- [ ] 逻辑是否简洁
- [ ] 是否有重复代码
- [ ] 注释是否充分

**安全性：**
- [ ] 是否有注入风险
- [ ] 敏感数据是否加密
- [ ] 权限检查是否完善

**性能：**
- [ ] 是否有不必要的循环
- [ ] 数据库查询是否优化
- [ ] 是否有内存泄漏风险

**可维护性：**
- [ ] 代码是否易于理解
- [ ] 是否易于扩展
- [ ] 依赖是否合理

### 3. Review 输出格式

```markdown
## Code Review 报告

**PR:** #XX
**审查人：** Dev Agent
**审查时间：** YYYY-MM-DD

### 概要
- 总体评价：✅ 通过 / ⚠️ 需修改 / ❌ 不通过
- 发现问题：X 个
- 建议改进：X 个

### 问题列表

#### 🔴 必须修改
1. **[文件:行号]** 问题描述
   - 代码：`...`
   - 建议：`...`

#### 🟡 建议改进
1. **[文件:行号]** 改进建议
   - 当前：`...`
   - 建议：`...`

#### 🟢 优点
1. 做得好的地方...

### 总体建议
[整体改进方向]
```

### 4. Review 反馈原则

**建设性反馈：**
- ✅ "建议使用 === 而不是 ==，避免类型转换问题"
- ❌ "这里写错了"

**具体明确：**
- ✅ "第 15 行的循环可以优化为 filter + map"
- ❌ "性能不好"

**解释原因：**
- ✅ "建议提取为函数，便于复用和测试"
- ❌ "这里要改"

## 输出位置

```
workspace-dev/docs/code-review-YYYY-MM-DD.md
```

## 下一步

Review 通过后，调用 `dev-git-operations` 提交代码。
