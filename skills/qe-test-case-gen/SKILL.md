# QE Agent - 测试用例生成技能

## 概述

本技能指导 QE Agent 根据需求文档、User Story 和验收标准，设计全面、结构化的测试用例。好的测试用例设计需要考虑正常路径、异常路径、边界条件和各种业务场景，确保产品质量有充分的测试覆盖。

## 触发条件

- 收到新的 User Story 需要设计测试用例
- 需要对功能模块进行测试设计
- 收到"测试用例"、"测试设计"、"test case"、"设计测试"等指令
- Sprint Planning 后需要为即将开发的 Story 准备测试用例
- 需要补充现有功能的测试覆盖

## 流程

### 1. 需求分析

- 仔细阅读 User Story 和验收标准
- 梳理功能点和业务规则
- 识别输入参数、输出结果和状态变化
- 分析约束条件和边界值
- 与 PO 或 Dev 澄清不明确的细节

### 2. 测试场景识别

使用以下方法全面识别测试场景：

- **等价类划分**: 将输入分为有效和无效等价类
- **边界值分析**: 测试边界和临界值（如 0、-1、最大值、最小值）
- **状态转换测试**: 测试对象在不同状态间的转换
- **错误推测**: 基于经验推测可能的错误场景
- **场景分析**: 基于用户使用场景设计测试

### 3. 编写测试用例

每个测试用例包含：
```
**用例编号**: TC-XXX-001
**用例标题**: 简明描述测试场景
**前置条件**: 执行测试前的准备条件
**测试步骤**: 详细的操作步骤
**测试数据**: 需要的输入数据
**预期结果**: 每个步骤的预期输出
**优先级**: P0(必须) / P1(重要) / P2(一般) / P3(可选)
**关联需求**: 对应的 User Story 或验收标准
```

### 4. 测试用例评审

- 与 PO 确认测试用例是否覆盖所有验收标准
- 与 Dev 确认测试场景是否遗漏关键路径
- 检查用例的可执行性和清晰度
- 评审后根据反馈调整用例

### 5. 用例管理与维护

- 将用例录入测试管理工具
- 按模块和功能分类组织
- 随需求变更及时更新用例
- 定期清理过时的用例

## 输入

- **User Story**: 含验收标准的用户故事
- **需求文档**: 详细的功能规格说明
- **业务规则**: 相关的业务约束和逻辑规则
- **接口定义**: API 接口文档（如有）

## 输出

- **测试用例集**: 结构化的测试用例文档
- **测试矩阵**: 功能点与测试用例的覆盖矩阵
- **评审记录**: 用例评审的反馈和修改记录

## 注意事项

- **覆盖验收标准是底线**：每个验收标准至少有一条对应的测试用例。
- **关注异常场景**：异常路径往往隐藏最多的 Bug，不要只测 Happy Path。
- **用例要可执行**：步骤要具体到可操作，数据要具体可构造。
- **测试数据很重要**：准备好有效的测试数据，不要用随意的数据。
- **用例不是越多越好**：追求有效覆盖，而非数量。冗余用例增加维护成本。
- **评审不可省略**：一个人的思维有盲区，评审能显著提高用例质量。
