# 需求拆解 Skill

将产品愿景或用户需求转化为规范的 User Story。

## 工作流程

### 1. 理解需求

- 识别用户角色（谁需要这个功能）
- 理解业务目标（为什么要做）
- 确认功能范围（做什么）
- 追问澄清模糊点

### 2. 拆解原则

**INVEST 原则：**
- **I**ndependent — 独立，不依赖其他 Story
- **N**egotiable — 可协商，细节可以讨论
- **V**aluable — 有价值，对用户有意义
- **E**stimable — 可估算，团队能评估工作量
- **S**mall — 足够小，一个 Sprint 能完成
- **T**estable — 可测试，有明确的验收标准

**拆解粒度：**
- 一个 Story = 一个完整用户价值
- 工作量：1-5 故事点为佳
- 超过 8 故事点考虑继续拆分

### 3. 输出格式

```markdown
## US-XXX: [故事标题]

**作为** [用户角色]
**我想要** [功能描述]
**以便于** [价值说明]

### 验收标准
- [ ] Given [前置条件] When [操作] Then [预期结果]
- [ ] ...

### 优先级
- [ ] P0 - 必须有
- [ ] P1 - 应该有
- [ ] P2 - 可以有
- [ ] P3 - 不会有

### 故事点估算
- 估算：[ ] 1  [ ] 2  [ ] 3  [ ] 5  [ ] 8

### 依赖
- 前置依赖：US-XXX（如有）
- 后置依赖：US-XXX（如有）

### 备注
[补充说明]
```

### 4. 验证检查

拆解完成后，自检：
- [ ] 每个 Story 是否符合 INVEST 原则？
- [ ] 验收标准是否可测试？
- [ ] 是否有遗漏的场景？
- [ ] Story 之间是否独立？

## 常见拆解模式

### 按用户角色拆解
```
需求：用户登录
→ US-001: 普通用户登录
→ US-002: 管理员登录
→ US-003: 第三方账号登录
```

### 按功能维度拆解
```
需求：商品搜索
→ US-001: 关键词搜索
→ US-002: 分类筛选
→ US-003: 排序功能
→ US-004: 搜索历史
```

### 按优先级拆解
```
需求：用户中心
→ US-001: 查看个人信息（P0）
→ US-002: 修改个人信息（P1）
→ US-003: 头像上传（P2）
```

## 输出位置

将 User Story 保存到：
```
workspace-po/backlog/user-stories/US-XXX.md
```

## 下一步

拆解完成后，调用 `po-backlog-management` 进行优先级排序。
