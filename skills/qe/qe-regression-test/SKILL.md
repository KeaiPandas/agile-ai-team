# 回归测试

## 概述

QE Agent 在 Bug 修复后执行回归测试，验证修复有效且未引入新问题。回归测试包括：针对修复 Bug 的验证测试、相关功能的回归测试、全量测试（里程碑发布前）。

**在流程中的位置：** Bug 修复 → **回归测试** → 确认发布

## 触发条件

- Bug 报告状态变为 `fixed`，需要验证修复
- 里程碑发布前执行全量回归测试
- 代码合并到 main 分支后（CI 触发）

## 输入

- `bugs/BUG-XXX.md`（被修复的 Bug 详情）
- 全部自动化测试用例
- `sprint/tasks/T-XXX/test-plan.md`（相关 Task 的测试计划）
- 修复后的代码

## 流程

1. **验证修复有效性**：
   - 运行 Bug 对应的失败测试用例
   - 确认之前失败的用例现在通过
   - 手动验证复现步骤不再触发 Bug
2. **执行影响范围回归**：
   - 运行与 Bug 相关模块的所有测试用例
   - 运行依赖该模块的下游模块测试
   - 确认相关功能未受修复影响
3. **执行全量回归（发布前）**：
   - 运行项目全部自动化测试
   - 检查测试通过率是否与修复前一致
   - 确认没有新增失败用例
4. **分析结果**：
   - 全部通过 → Bug 修复验证通过，更新 Bug 状态为 `verified`
   - 新增失败 → 修复引入了回归，创建新 Bug 报告，退回开发
   - 原有 Bug 未修复 → 验证不通过，退回开发重新修复
5. **更新状态**：更新 Bug 报告和 `sprint-status.md`

## 输出

- 回归测试报告（`sprint/tasks/T-XXX/regression-report.md`）
- 更新后的 Bug 状态
- 更新后的 `sprint-status.md`

## 文件规范

### 回归测试报告格式

路径：`sprint/tasks/T-XXX/regression-report.md`

```markdown
# T-XXX 回归测试报告

**执行时间:** YYYY-MM-DD HH:MM
**触发原因:** BUG-001 修复验证

## 修复验证
| Bug ID | 对应用例 | 修复前 | 修复后 | 结果 |
|--------|----------|--------|--------|------|
| BUG-001 | TC-003 | ❌ 失败 | ✅ 通过 | 已修复 |

## 回归测试
- **测试范围:** auth 模块全部用例 + 用户相关模块
- **用例总数:** 15
- **通过:** 15
- **失败:** 0
- **回归结果:** ✅ 无回归

## 结论
✅ BUG-001 修复验证通过，无回归问题。
```

## 注意事项

- **修复验证优先**：先确认 Bug 本身已修复，再检查回归
- **影响范围评估要全面**：不要只跑 Bug 对应的用例，相关模块也要覆盖
- **回归通过才能发布**：任何回归失败都应阻塞发布流程
- **记录回归结果**：每次回归都应有报告，形成质量追溯链
- **全量回归在发布前执行**：日常开发中只需跑影响范围内的回归
