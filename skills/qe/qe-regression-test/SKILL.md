# QE Agent - 回归测试技能

## 概述

本技能指导 QE Agent 执行回归测试。回归测试的目的是在代码变更（Bug 修复、功能新增、重构等）后，验证已有功能仍然正常工作，确保变更没有引入新的缺陷。回归测试是软件质量保障的最后一道防线。

## 触发条件

- Bug 修复后需要验证
- 新功能上线前需要确认无回归
- 代码重构后需要验证功能完整性
- 收到"回归测试"、"验证"、"regression"、"回归"等指令
- 版本发布前的质量门禁
- Sprint 结束后的交付验证

## 流程

### 1. 确定回归范围

根据变更的影响范围，确定回归测试的覆盖面：
- **全量回归**: 重大版本发布或大面积重构时执行
- **影响范围回归**: 基于代码变更分析，测试受影响的模块
- **冒烟回归**: 快速验证核心功能是否正常（15-30 分钟）

使用影响分析方法：
- 分析变更涉及的模块和接口
- 识别依赖这些模块的下游功能
- 确定需要回归的测试用例集

### 2. 执行回归测试

按优先级执行：
1. **验证修复的 Bug**: 确认原 Bug 已修复
2. **冒烟测试**: 验证核心流程（登录、支付、数据提交等）
3. **受影响模块测试**: 测试与变更相关的功能
4. **通用功能测试**: 搜索、导航、权限等通用功能
5. **全面回归**: 如有需要，执行完整的测试用例集

### 3. 记录结果

- 记录每个测试用例的执行结果（通过/失败/跳过）
- 对失败的用例进行详细分析
- 区分"新引入的 Bug"和"已有的 Bug"
- 截图记录失败现象

### 4. 回归结果处理

- **全部通过**: 确认质量达标，可以进入下一步（发布/合并）
- **发现新 Bug**: 提交 Bug 报告，评估是否阻塞发布
- **发现已有 Bug**: 评估严重程度，决定是否需要修复后才能发布

### 5. 回归测试报告

- 测试范围和执行摘要
- 通过/失败统计
- 发现的问题清单及严重程度
- 发布建议（Go/No-Go）

## 输入

- **变更信息**: Bug 修复记录或代码变更说明
- **影响分析**: 变更影响的模块和功能
- **测试用例库**: 已有的测试用例集
- **测试环境**: 部署了最新代码的测试环境

## 输出

- **回归测试报告**: 执行结果和统计分析
- **新 Bug 清单**: 回归过程中发现的新缺陷
- **发布建议**: Go / No-Go 的质量评估结论

## 注意事项

- **先验证修复再回归**: 确认修复的 Bug 确实解决了，再进行更广泛的回归。
- **回归范围要合理**: 不要每次都全量回归，根据变更影响范围灵活调整。
- **自动化优先**: 尽可能将回归测试自动化，提高效率和一致性。
- **关注性能回归**: 除了功能回归，也要关注性能是否退化。
- **回归测试要有时间盒**: 控制回归测试的时间，避免无限扩展。
- **记录回归发现的价值**: 回归测试发现新 Bug 说明回归有价值，用数据证明质量保障的价值。
