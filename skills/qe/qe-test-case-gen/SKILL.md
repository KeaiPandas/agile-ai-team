# 测试用例生成

## 概述

QE Agent 根据 Task 的验收标准自动生成测试用例。测试用例覆盖正常流程、边界条件和异常场景，确保 Task 实现的质量。测试用例是自动化测试的基础，也是验证 Task 是否达到交付标准的依据。

**在流程中的位置：** 代码审查通过 → **测试用例生成** → 自动化测试

## 触发条件

- `sprint-status.md` 中有 Task 的状态变为 `review` 且分配给 `qe`
- 里程碑启动时预生成测试用例（Shift-Left）
- 收到新的 Bug 需要补充回归测试用例

## 输入

- `backlog/tasks/T-XXX.md`（Task 描述和验收标准）
- 被测代码（了解实现细节以设计用例）
- `sprint/tasks/T-XXX/work-log.md`（开发记录，了解实现方案）
- 现有测试文件（了解测试风格和工具）

## 流程

1. **分析验收标准**：逐条拆解验收标准，提取可测试的断言点
2. **设计测试用例**：
   - **正常流程（Happy Path）**：验收标准中描述的正常使用场景
   - **边界条件**：空输入、最大值、最小值、特殊字符
   - **异常场景**：网络错误、权限不足、数据不存在
   - **兼容性**：不同浏览器/平台/数据格式
3. **确定测试类型**：
   - 单元测试：函数、方法级别的测试
   - 集成测试：模块间交互的测试
   - 端到端测试：用户操作流程的测试
4. **编写测试用例文件**：使用项目约定的测试框架和风格
5. **标记优先级**：
   - **关键（Critical）**：必须通过，对应 P0 验收标准
   - **重要（Major）**：应该通过，对应 P1 场景
   - **一般（Minor）**：边缘情况，对应 P2 场景
6. **输出用例清单**：写入 `sprint/tasks/T-XXX/test-plan.md`

## 输出

- 测试用例文件（代码）
- `sprint/tasks/T-XXX/test-plan.md`（测试计划清单）

## 文件规范

### 测试计划格式

路径：`sprint/tasks/T-XXX/test-plan.md`

```markdown
# T-XXX 测试计划

## 测试范围
- 功能: xxx
- 模块: xxx

## 测试用例

### TC-001: 正常登录
- **优先级:** Critical
- **类型:** 单元测试
- **前置条件:** 用户已注册
- **步骤:** 调用 login API with valid credentials
- **预期:** 返回 token 和用户信息，状态码 200

### TC-002: 密码错误
- **优先级:** Critical
- **类型:** 单元测试
- **前置条件:** 用户已注册
- **步骤:** 调用 login API with wrong password
- **预期:** 返回错误信息，状态码 401

### TC-003: 空邮箱输入
- **优先级:** Major
- **类型:** 单元测试
- **步骤:** 调用 login API with empty email
- **预期:** 返回验证错误，状态码 422

## 统计
- Critical: 2 | Major: 3 | Minor: 2 | 总计: 7
```

## 注意事项

- **验收标准全覆盖**：每条验收标准至少有一个对应的测试用例
- **边界条件不能忽略**：Bug 往往出在边界条件
- **用例独立**：每个测试用例不依赖其他用例的执行结果
- **用例可重复**：相同输入必须得到相同输出
- **测试文件位置遵循项目规范**：与被测代码保持对应关系
