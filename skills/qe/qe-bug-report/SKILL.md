# Bug 报告

## 概述

QE Agent 在测试过程中发现缺陷时，创建标准化的 Bug 报告。Bug 报告包含完整的复现步骤、期望和实际行为、严重程度等信息，是开发修复和质量跟踪的依据。每个 Bug 有唯一 ID，便于跟踪和管理。

**在流程中的位置：** 自动化测试发现缺陷 → **创建 Bug 报告** → Dev 修复 → 回归测试验证

## 触发条件

- 自动化测试发现失败的测试用例（排除测试自身问题）
- 手动测试中发现缺陷
- 生产环境反馈的缺陷

## 输入

- 失败的测试用例详情
- 被测代码和错误日志
- 相关 Task 信息（`T-XXX`）
- 环境信息（操作系统、浏览器、版本等）

## 流程

1. **确认是真实 Bug**：
   - 排除测试用例自身问题（断言错误、环境问题）
   - 确认不是预期行为（对照验收标准）
2. **收集 Bug 信息**：
   - 错误信息和堆栈跟踪
   - 复现步骤
   - 期望行为 vs 实际行为
   - 影响范围
3. **评估严重程度**：
   - **严重**：系统崩溃、数据丢失、安全漏洞
   - **一般**：功能异常、用户体验受损
   - **轻微**：UI 显示问题、非关键路径异常
4. **创建 Bug 报告**：写入 `bugs/` 目录
5. **通知相关 Agent**：通知 Dev Agent 有新 Bug 需要修复
6. **更新 Task 状态**：如有 Bug，相关 Task 状态标记为 `blocked`

## 输出

- Bug 报告文件（`bugs/BUG-XXX.md`）
- Bug 汇总更新（`bugs/bug-summary.md`）
- 更新后的 `sprint-status.md`

## 文件规范

### Bug 报告格式

路径：`bugs/BUG-XXX.md`

```markdown
## BUG-001: <标题>

- **严重程度:** 严重 / 一般 / 轻微
- **相关 Task:** T-001
- **发现时间:** YYYY-MM-DD HH:MM
- **发现者:** qe
- **状态:** open / fixed / verified

### 描述
简要描述 Bug 的现象。

### 复现步骤
1. 步骤 1
2. 步骤 2
3. 步骤 3

### 期望行为
应该发生什么。

### 实际行为
实际发生了什么。

### 错误信息
```
错误日志或堆栈跟踪
```

### 环境信息
- OS: xxx
- 版本: xxx
- 浏览器: xxx（如适用）

### 修复记录
- **修复者:** dev
- **修复时间:** YYYY-MM-DD HH:MM
- **修复方式:** 简述修复方案

### 验证记录
- **验证者:** qe
- **验证时间:** YYYY-MM-DD HH:MM
- **验证结果:** 通过 / 未通过
```

### Bug 汇总格式

路径：`bugs/bug-summary.md`

```markdown
# Bug 汇总

## 当前里程碑
| Bug ID | 标题 | 严重程度 | 相关 Task | 状态 |
|--------|------|----------|-----------|------|
| BUG-001 | xxx | 严重 | T-001 | open |
| BUG-002 | xxx | 一般 | T-002 | fixed |
```

## 注意事项

- **Bug 报告必须可复现**：复现步骤必须精确，其他人能按照步骤重现问题
- **严重程度判断准确**：严重 Bug 优先处理，避免资源错配
- **不要重复报告**：创建前检查 `bugs/bug-summary.md` 是否已有相同 Bug
- **状态及时更新**：修复后立即更新 Bug 状态，闭环管理
- **保留历史记录**：已验证的 Bug 保留记录，不删除
