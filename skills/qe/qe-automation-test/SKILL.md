# 自动化测试

## 概述

QE Agent 编写并执行自动化测试脚本，验证 Task 实现是否满足验收标准。自动化测试包括单元测试、集成测试和端到端测试，是质量门禁的核心组成部分。测试通过后 Task 才能进入发布流程。

**在流程中的位置：** 测试用例生成 → **自动化测试执行** → Bug 报告 / 通过

## 触发条件

- 测试用例生成完毕后自动执行
- 代码变更后需要重新运行相关测试（CI 触发）
- Bug 修复后需要验证修复

## 输入

- 测试用例文件（来自测试用例生成步骤）
- 被测代码（最新版本）
- 测试环境配置
- `sprint/tasks/T-XXX/test-plan.md`（测试计划）

## 流程

1. **准备测试环境**：
   - 确认依赖服务（数据库、缓存、第三方 API mock）已就绪
   - 配置测试环境变量
   - 初始化测试数据
2. **编写测试脚本**：
   - 基于测试计划中的用例编写可执行的测试代码
   - 使用项目约定的测试框架（Jest、Vitest、Playwright 等）
   - 添加必要的 setup/teardown 逻辑
3. **执行测试**：
   - 运行全部测试用例
   - 收集测试结果（通过/失败/跳过）
   - 收集覆盖率数据
4. **分析结果**：
   - 全部通过 → Task 质量达标，更新状态为 `done`
   - 部分失败 → 分析失败原因，区分是 Bug 还是测试用例问题
   - 覆盖率不足 → 补充测试用例
5. **生成测试报告**：输出详细的测试结果和覆盖率报告
6. **更新状态**：将测试结果写入状态文件

## 输出

- 测试脚本文件
- 测试报告（`sprint/tasks/T-XXX/test-report.md`）
- 覆盖率报告
- 更新后的 `sprint-status.md`

## 文件规范

### 测试报告格式

路径：`sprint/tasks/T-XXX/test-report.md`

```markdown
# T-XXX 测试报告

**执行时间:** YYYY-MM-DD HH:MM
**执行者:** qe

## 结果概览
- ✅ 通过: 5
- ❌ 失败: 1
- ⏭️ 跳过: 0
- **通过率:** 83.3%

## 覆盖率
| 类型 | 覆盖率 |
|------|--------|
| 语句 | 92% |
| 分支 | 85% |
| 函数 | 100% |
| 行 | 90% |

## 失败用例

### ❌ TC-003: 空邮箱输入
- **文件:** `tests/auth/login.test.ts:42`
- **错误:** Expected status 422, received 200
- **原因:** 后端未校验空邮箱

## 结论
⚠️ 不通过 — 1 个用例失败，已创建 Bug 报告
```

## 注意事项

- **测试环境隔离**：测试不应影响其他环境和其他测试
- **确定性**：测试结果必须确定，不应因时序或随机因素产生 flaky test
- **执行速度**：单元测试应快速完成，端到端测试可以稍慢但不超 5 分钟
- **覆盖率是参考**：高覆盖率不等于高质量，但要确保关键路径有覆盖
- **Mock 外部依赖**：不要依赖真实的外部服务，使用 mock 替代
