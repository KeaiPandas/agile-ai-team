# Scrum Master - 障碍清除技能

## 概述

本技能指导 Scrum Master 识别、记录和协调解决 Sprint 中出现的各种阻塞问题。Blocker 是阻碍团队高效交付的障碍，Scrum Master 的核心职责之一就是"扫清障碍"，让开发团队能够专注于创造价值。优秀的 Scrum Master 不仅解决问题，更要预防问题的发生。

## 触发条件

- 团队成员报告遇到阻塞
- 发现进度停滞或延期
- 收到"障碍"、"阻塞"、"blocker"、"清除障碍"、"卡住了"等指令
- 站会中发现有成员被阻塞
- 跨团队依赖导致协作问题
- 环境或工具问题影响工作效率

## 流程

### 1. 识别 Blocker

Blocker 的常见类型：
- **依赖阻塞**: 等待其他团队或人员的输入/接口/审批
- **技术阻塞**: 技术难题、环境问题、工具故障
- **需求阻塞**: 需求不清晰、验收标准有歧义
- **资源阻塞**: 缺少必要的资源、权限或环境
- **流程阻塞**: 审批流程过长、沟通效率低

### 2. 记录 Blocker

使用统一格式记录：
```
**Blocker 编号**: BLK-001
**描述**: 简要描述阻塞问题
**影响范围**: 被阻塞的任务/人员
**严重程度**: 高/中/低
**提出人**: 发现阻塞的团队成员
**提出时间**: 发现阻塞的时间
**负责人**: 负责解决的人
**当前状态**: 处理中/已解决/待升级
```

### 3. 评估和分级

- **P0 - 紧急**: 团队完全无法工作（如服务器宕机、构建环境损坏）
- **P1 - 高**: 单个成员完全阻塞，无法推进（如缺少关键接口）
- **P2 - 中**: 部分受阻，可做其他工作（如等待设计评审）
- **P3 - 低**: 轻微影响，不影响核心进度（如工具使用不便）

### 4. 协调解决

- **内部解决**: 协调团队内部资源快速解决
- **跨团队沟通**: 与相关团队沟通协调，推动问题解决
- **升级处理**: 超出自身能力范围时，及时向上升级
- **临时替代方案**: 在等待彻底解决期间，寻找临时替代方案让工作可以继续

### 5. 跟踪和关闭

- 每日跟踪 Blocker 处理进度
- 解决后确认团队可以继续推进
- 记录解决方案和经验教训
- 分析 Blocker 的根本原因，制定预防措施

## 输入

- **Blocker 报告**: 团队成员报告的阻塞问题
- **Sprint 进度**: 当前 Sprint 的进度和状态
- **团队反馈**: 站会、聊天中的障碍反馈
- **依赖关系**: 任务间的依赖和外部依赖

## 输出

- **Blocker 清单**: 当前活跃的阻塞问题列表
- **解决记录**: 问题解决过程和结果
- **预防建议**: 防止同类问题再次发生的措施

## 注意事项

- **快速响应**：Blocker 越早处理，对进度的影响越小。不要等积压多了再处理。
- **不要自己扛**：超出能力范围的 Blocker 要及时升级，不要硬扛导致问题扩大。
- **关注根本原因**：解决 Blocker 后要分析原因，预防同类问题再次发生。
- **让团队专注**：Scrum Master 的核心价值就是让开发团队不被打扰地工作。
- **主动发现**：不要只等团队报告 Blocker，要主动在站会和日常沟通中识别潜在障碍。
- **记录 Blocker 趋势**：定期分析 Blocker 的类型和频率，发现系统性问题。
