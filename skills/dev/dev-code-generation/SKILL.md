# 代码生成

## 概述

Dev Agent 从 `sprint-status.md` 领取待执行的 Task，读取 Task 描述和验收标准，生成符合要求的代码实现。代码生成遵循项目现有架构风格，确保可测试、可维护。

**在流程中的位置：** 里程碑启动 → **代码生成** → 代码审查

## 触发条件

- `sprint-status.md` 中有分配给 `dev` 的 `todo` 状态 Task
- 代码审查反馈需要修改

## 输入

- `backlog/tasks/T-XXX.md`（Task 描述和验收标准）
- 项目现有代码（了解架构风格、技术栈、命名规范）
- 项目配置文件（`package.json`、`tsconfig.json` 等）
- 相关技术文档

## 流程

1. **领取 Task**：从 `sprint-status.md` 中找到分配给自己的 `todo` 状态 Task
2. **理解需求**：仔细阅读 Task 描述和验收标准，确保完全理解
3. **分析现有代码**：
   - 了解项目架构和目录结构
   - 找到需要修改/新增的位置
   - 识别相关模块和依赖关系
4. **设计实现方案**：
   - 确定代码组织方式（新文件/修改现有文件）
   - 规划数据流和接口设计
   - 考虑边界情况和错误处理
5. **生成代码**：
   - 遵循项目现有的代码风格和命名规范
   - 添加必要的注释（复杂逻辑处）
   - 处理边界情况和错误场景
   - 确保类型安全（TypeScript 项目）
6. **自测验证**：
   - 静态检查（lint、类型检查）
   - 基本功能验证
   - 对照验收标准逐条检查
7. **更新状态**：
   - 将 `sprint-status.md` 中 Task 状态改为 `review`
   - 更新时间戳
8. **记录工作日志**：在 Task 目录下创建工作记录

## 输出

- 代码文件（新增或修改）
- 更新后的 `sprint-status.md`
- Task 工作记录（`sprint/tasks/T-XXX/work-log.md`）

## 文件规范

### 工作记录格式

路径：`sprint/tasks/T-XXX/work-log.md`

```markdown
# T-XXX 工作记录

## 基本信息
- **执行者:** dev
- **开始时间:** YYYY-MM-DD HH:MM
- **完成时间:** YYYY-MM-DD HH:MM

## 实现概要
简要描述实现方案。

## 修改文件
- `src/auth/login.ts` — 新增登录逻辑
- `src/types/user.ts` — 新增 User 类型定义

## 验收标准检查
- [x] 标准 1
- [x] 标准 2
- [x] 标准 3

## 备注
遇到的问题和解决方案。
```

## 注意事项

- **先理解再动手**：不要跳过需求分析直接写代码
- **遵循项目风格**：新代码必须与现有代码风格一致
- **处理边界情况**：不要只写 happy path
- **验收标准是硬约束**：每条验收标准都必须满足
- **代码可读性优先**：Agent 生成的代码也会被其他 Agent 阅读
