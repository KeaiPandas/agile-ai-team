# Dev Agent - Code Review 技能

## 概述

本技能指导 Dev Agent 对团队成员提交的代码进行系统性的审查。Code Review 不仅仅是"看看代码写得怎么样"，更是保障代码质量、传播最佳实践、降低技术债务的重要环节。Dev Agent 应以建设性的态度进行审查，既指出问题也认可优秀的实现。

## 触发条件

- 收到 PR（Pull Request）或 MR（Merge Request）
- 收到"code review"、"代码审查"、"审查代码"、"review"等指令
- 团队成员请求代码审查
- Sprint 结束前需要进行代码质量检查
- 合并到主分支前的质量门禁

## 流程

### 1. 准备工作

- 阅读 PR 描述，理解变更的目的和背景
- 查看关联的 User Story 或 Issue
- 确认变更范围是否与描述一致

### 2. 整体审查

- **变更范围检查**: PR 是否过大？是否应该拆分为多个小 PR？
- **架构一致性**: 变更是否符合项目的整体架构？
- **设计合理性**: 方案是否合理？有无更优的替代方案？

### 3. 详细审查

按以下维度逐一检查：

- **正确性**: 逻辑是否正确？是否覆盖边界条件？
- **代码质量**: 可读性如何？命名是否清晰？是否有重复代码？
- **性能**: 是否有明显的性能问题？N+1 查询、不必要的循环等？
- **安全性**: 是否存在安全漏洞？输入校验、权限控制是否到位？
- **可维护性**: 代码是否易于理解和修改？耦合度是否合理？
- **测试覆盖**: 是否有对应的测试？测试是否充分？
- **代码风格**: 是否符合项目编码规范？

### 4. 编写 Review 意见

- 按严重程度分类：🔴 阻塞（必须修复）/ 🟡 建议（推荐修复）/ 🟢 点赞（优秀实践）
- 每条意见都应包含：问题描述 + 具体位置 + 改进建议（最好附代码示例）
- 语气建设性，避免指责或嘲讽
- 认可优秀的代码实现

### 5. 总结反馈

- 给出整体评价和合并建议（Approve / Request Changes / Comment）
- 列出关键改进项
- 如有重大问题，建议线下讨论

## 输入

- **PR/MR**: 包含代码变更的 Pull Request
- **关联需求**: 对应的 User Story 或 Issue
- **代码上下文**: 项目整体代码库（用于理解变更的影响范围）

## 输出

- **Review 意见**: 结构化的代码审查意见列表
- **整体评价**: Approve / Request Changes / Comment
- **改进建议**: 具体的优化方向和代码示例

## 注意事项

- **关注问题而非人**：Review 针对代码，不是针对编写者。保持专业和尊重。
- **不要吹毛求疵**：区分"必须修改"和"可以更好"，不要让 minor 风格问题阻塞合并。
- **及时响应**：收到 Review 请求后尽快处理，避免成为团队瓶颈。
- **平衡速度与质量**：既要确保质量，也要尊重交付节奏。
- **学习与分享**：Review 是双向学习的机会，从他人代码中学到好的实践。
- **关注变更本身**：不要借 Review 之机重构不相关的代码。
- **提供可操作的建议**：不只是指出问题，要给出解决方案。
