# 代码审查

## 概述

Dev Agent 对其他 Agent 提交的代码进行交叉审查。审查重点包括代码质量、安全性、性能、可维护性和对验收标准的符合度。审查通过后 Task 才能进入 QE 测试环节。

**在流程中的位置：** 代码生成 → **代码审查** → QE 测试

## 触发条件

- `sprint-status.md` 中有 `review` 状态的 Task
- 代码提交后自动触发（通过 Git Hook 或 CI）

## 输入

- 被审查的代码变更（Git diff 或 PR）
- `backlog/tasks/T-XXX.md`（Task 描述和验收标准，用于对照）
- 项目代码规范（如有 ESLint、Prettier 配置）
- `sprint/tasks/T-XXX/work-log.md`（开发者的工作记录）

## 流程

1. **领取审查任务**：从 `sprint-status.md` 找到 `review` 状态的 Task
2. **阅读 Task 上下文**：理解 Task 目标和验收标准
3. **审查代码变更**：
   - **正确性**：逻辑是否正确，是否满足验收标准
   - **安全性**：是否存在注入、XSS、认证绕过等安全漏洞
   - **性能**：是否存在明显性能问题（N+1 查询、内存泄漏等）
   - **可维护性**：命名是否清晰、职责是否单一、是否有过度抽象
   - **代码风格**：是否符合项目规范
   - **错误处理**：边界情况和异常是否被妥善处理
   - **测试覆盖**：是否有必要的单元测试
4. **撰写审查意见**：
   - 必须修复（blocking）：逻辑错误、安全漏洞
   - 建议修改（suggestion）：性能优化、代码风格
   - 赞许（praise）：优秀的实现方式
5. **给出结论**：
   - **通过**：代码质量达标，推进到 QE 测试
   - **需修改**：存在必须修复的问题，退回开发
6. **更新状态**：修改 `sprint-status.md` 中 Task 状态

## 输出

- 代码审查报告（写入 `sprint/tasks/T-XXX/review.md`）
- 更新后的 `sprint-status.md`

## 文件规范

### 审查报告格式

路径：`sprint/tasks/T-XXX/review.md`

```markdown
# T-XXX 代码审查报告

## 基本信息
- **审查者:** dev
- **审查时间:** YYYY-MM-DD HH:MM
- **结论:** 通过 / 需修改

## 审查意见

### ❌ 必须修复 (Blocking)
1. `src/auth/login.ts:42` — SQL 拼接存在注入风险，改用参数化查询

### ⚠️ 建议修改 (Suggestion)
1. `src/auth/login.ts:15` — 魔法字符串建议提取为常量

### ✅ 亮点
1. 错误处理覆盖全面，包含了各种异常场景

## 验收标准符合度
- [x] 标准 1 — 通过
- [x] 标准 2 — 通过
- [ ] 标准 3 — 未完全覆盖，缺少 XXX 场景
```

## 注意事项

- **审查者不能审查自己的代码**：确保交叉审查的有效性
- **关注安全性**：AI Agent 生成的代码同样需要安全审查
- **结论明确**：通过或需修改，不含糊
- **意见具体**：指出具体的文件、行号和问题，避免笼统描述
- **尊重开发者的设计选择**：建议性的问题不阻塞，但必须修复的问题坚决退回
