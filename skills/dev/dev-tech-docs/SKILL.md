# Dev Agent - 技术文档生成技能

## 概述

本技能指导 Dev Agent 编写和维护各类技术文档。好的技术文档是团队协作的基石——它降低新成员的学习成本，减少重复沟通，让知识不因人员变动而流失。Dev Agent 应将文档视为代码的一部分，做到"代码即文档，文档即代码"。

## 触发条件

- 需要为新功能或模块编写技术文档
- 需要更新 API 文档
- 收到"写文档"、"技术文档"、"API 文档"、"文档更新"等指令
- 新成员加入需要 onboarding 文档
- 项目架构发生变化需要更新文档
- Sprint Review 后需要更新技术文档

## 流程

### 1. 确定文档类型和受众

常见文档类型：
- **API 文档**: 接口定义、请求/响应格式、示例
- **架构文档**: 系统架构、模块关系、技术选型
- **开发指南**: 环境搭建、开发流程、编码规范
- **部署文档**: 部署流程、环境配置、运维手册
- **决策记录**: ADR (Architecture Decision Records)

明确文档的目标读者：开发者、运维、测试、还是产品？

### 2. 收集信息

- 阅读相关代码和配置
- 与相关团队成员沟通确认细节
- 查阅已有的文档，避免重复和冲突
- 收集使用示例和最佳实践

### 3. 编写文档

- **结构清晰**: 使用目录、标题、列表等组织内容
- **示例丰富**: 提供具体的使用示例和代码片段
- **保持同步**: 文档与代码实现保持一致
- **语言简洁**: 用简洁明了的语言，避免歧义
- **版本标注**: 标注适用的版本号和更新日期

### 4. 审查和发布

- 请团队成员审查文档的准确性
- 确认文档放置在正确位置（如 `/docs` 目录）
- 更新文档索引或目录

### 5. 持续维护

- 代码变更时同步更新相关文档
- 定期检查文档是否过时
- 收集文档使用反馈并持续改进

## 输入

- **代码实现**: 需要文档化的代码或模块
- **需求上下文**: 功能背景和使用场景
- **团队规范**: 文档格式、存放位置、风格要求
- **现有文档**: 需要更新或补充的已有文档

## 输出

- **技术文档**: Markdown/AsciiDoc 格式的文档文件
- **API 参考**: 接口定义和示例
- **架构图**: 系统架构说明（文字或图表描述）

## 注意事项

- **文档不要滞后于代码**：代码改了，文档要同步更新。过时的文档比没有文档更糟糕。
- **写给人看，不是写给机器**：文档应该让不熟悉代码的人也能理解。
- **善用示例**：一个具体的示例胜过千言万语的描述。
- **保持文档 DRY**：避免在多处重复相同内容，用引用和链接代替。
- **文档也要 Review**：技术文档和代码一样，需要审查和迭代。
- **不要过度文档化**：不是所有代码都需要文档，关键接口和复杂逻辑优先。
