# 进度监控

## 概述

替代传统每日站会。Quality Agent 异步读取各 Agent 的工作状态文件（Task 工作记录、测试报告等）和 `sprint-status.md`，汇总当前里程碑进度，识别风险和阻塞项。通过状态文件驱动而非会议同步，Agent 随时可查看进度。

**在流程中的位置：** 里程碑启动 → **持续进度监控** → 阻塞清除（如有）

## 触发条件

- 定期检查（建议每完成一个 Task 后触发）
- Agent 更新状态文件后触发
- PO Agent 请求进度报告
- 发现 Task 长时间未更新（超过预期周期）

## 输入

- `sprint/sprint-status.md`（里程碑状态）
- `backlog/product-backlog.md`（Backlog 状态）
- 各 Agent 的工作记录文件（如有）
- `sprint/milestone-plan.md`（里程碑计划和成功标准）

## 流程

1. **读取里程碑状态**：加载 `sprint-status.md`，了解各 Task 当前状态
2. **扫描 Agent 工作文件**：检查是否有 Task 目录下的工作记录更新
3. **对比预期进度**：
   - 已完成 Task 数量 vs 计划 Task 总数
   - 各 Agent 的完成率
   - 是否有 Task 长时间停留在 `in-progress` 状态
4. **识别风险和阻塞**：
   - 超过预期时间 2 倍仍未完成的 Task → 标记为潜在阻塞
   - 状态为 `blocked` 的 Task → 列入阻塞清单
   - 依赖链中上游未完成 → 提醒下游 Agent 等待
5. **更新状态文件**：
   - 同步各文件中的最新状态到 `sprint-status.md`
   - 更新阻塞项列表
   - 更新时间戳
6. **输出进度报告**：生成简洁的进度摘要

## 输出

- 更新后的 `sprint/sprint-status.md`
- 进度报告（完成率、风险项、建议行动）

## 文件规范

### 进度报告格式（输出到日志或通知）

```
📊 里程碑进度报告 - <名称>
━━━━━━━━━━━━━━━━━━
完成: 3/8 Task (37.5%)
  ✅ T-001 用户登录 (dev)
  ✅ T-002 数据库设计 (dev)
  ✅ T-005 API 文档 (dev)
进行中: 2 Task
  🔄 T-003 首页开发 (dev) - 已进行 2h
  🔄 T-004 登录测试 (qe)
阻塞: 1 Task
  ⚠️ T-006 支付集成 - 等待第三方 API 文档
━━━━━━━━━━━━━━━━━━
下一步: dev 完成 T-003 后启动 T-007
```

## 注意事项

- **不做站会**：所有信息通过文件传递，Agent 自行读取状态文件即可
- **关注阻塞而非进度**：进度监控的核心价值是发现和解决阻塞
- **频率灵活**：不需要固定频率，按需检查即可
- **保持报告简洁**：Agent 不需要长篇报告，关键信息一目了然
- **尊重 Agent 自治**：只报告和建议，不强制干预 Agent 的执行方式
