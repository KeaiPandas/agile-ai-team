# 里程碑规划

## 概述

以可交付的里程碑（Milestone）为单位组织工作，替代传统的 Sprint Planning。里程碑以"可演示的交付物"为目标，不限定固定时间周期。PO Agent 根据 Backlog 优先级和当前进展，确定下一个里程碑的目标、范围和成功标准。

**在流程中的位置：** Backlog 管理 → **里程碑规划** → 里程碑启动（Scrum Master）

## 触发条件

- 当前里程碑接近完成（剩余 Task < 20%）
- 收到新的高优先级需求需要纳入下一个里程碑
- 项目启动时进行首次里程碑规划
- 外部因素导致需要调整里程碑目标

## 输入

- `backlog/product-backlog.md`（所有待办 Task）
- `sprint-status.md`（当前里程碑进展）
- 产品路线图（如有）
- 资源和约束条件

## 流程

1. **评估当前进展**：读取 `sprint-status.md`，确认当前里程碑的完成度
2. **回顾未完成 Task**：识别当前里程碑中未完成的 Task，决定是延后还是降级
3. **确定里程碑目标**：基于 Backlog 中高优先级 Task，定义一个可交付的目标
   - 目标必须是一句话可描述的、可演示的交付物
   - 例如："完成用户认证模块，支持邮箱登录和 OAuth"
4. **选择 Task 范围**：
   - 优先纳入所有相关的 P0 Task
   - 按依赖关系补充必要的 P1 Task
   - 总大小控制在合理范围（建议 3-5 个 M 等价的总量）
5. **定义成功标准**：列出里程碑完成的可验证条件
6. **输出里程碑计划**：写入 `sprint/milestone-plan.md`

## 输出

- `sprint/milestone-plan.md` — 里程碑计划文件
- 里程碑摘要（目标、范围、预估 Task 数量）

## 文件规范

### 里程碑计划格式

路径：`sprint/milestone-plan.md`

```markdown
# 里程碑计划 - <名称>

**创建日期:** YYYY-MM-DD
**状态:** planned / active / completed

## 目标
<一句话描述可交付的成果>

## 范围

### 包含的 Task
| ID | Task | Size | 优先级 |
|----|------|------|--------|
| T-001 | xxx | M | P0 |
| T-002 | xxx | S | P1 |

### 延后的 Task
| ID | Task | 原因 | 目标里程碑 |
|----|------|------|------------|
| T-003 | xxx | 依赖未满足 | M-002 |

## 成功标准
1. [ ] 标准 1
2. [ ] 标准 2

## 风险
- 风险项及应对策略

## 备注
补充信息
```

## 注意事项

- **里程碑粒度适中**：太小则规划频繁浪费资源，太大则失去聚焦
- **目标必须可演示**：每个里程碑结束后应该有可运行的、可展示的成果
- **不要贪多**：宁可少做做好，也不要堆砌半成品
- **里程碑之间允许重叠**：不像传统 Sprint 有硬边界，Task 可以跨里程碑
- **P0 优先**：永远先确保 P0 Task 被纳入，再考虑 P1/P2
