# PO Agent - Backlog 管理技能

## 概述

本技能指导 PO Agent 管理和优化产品待办列表（Product Backlog）。Backlog 是产品开发的核心输入，一个有序、清晰的 Backlog 能确保团队高效交付最大价值。PO Agent 需要持续维护 Backlog 的健康度，确保每个条目都有清晰的定义、合理的优先级和准确的估算。

## 触发条件

- 需要整理产品 Backlog
- 需要进行优先级排序
- 收到"管理 Backlog"、"优先级排序"、"整理待办"、"backlog"等指令
- 新的需求或反馈需要添加到 Backlog
- Sprint 结束后需要重新评估 Backlog
- 发现 Backlog 条目过时或不再相关

## 流程

### 1. 收集需求条目

- 从多个渠道收集需求：用户反馈、市场调研、内部需求、技术债务等
- 将收集到的需求初步整理为 Backlog 条目
- 每个条目包含：标题、描述、来源、业务价值

### 2. 分类与整理

按类型分类：
- 🆕 新功能 (Feature)
- 🐛 Bug 修复
- 🔧 技术改进
- 📖 文档更新
- 📊 性能优化
- 🔒 安全加固

### 3. 优先级排序

使用优先级评估框架（如 WSJF、MoSCoW、价值/努力矩阵）：
- **Must Have**: 必须交付，否则产品无法使用
- **Should Have**: 重要的增值功能
- **Could Have**: 锦上添花的功能
- **Won't Have**: 当前版本不做

考虑因素：
- 业务价值和用户影响
- 紧急程度
- 技术依赖关系
- 风险和不确定性

### 4. 条目细化

- 将高优先级的粗粒度条目细化为 User Story
- 确保每个条目都有验收标准（Acceptance Criteria）
- 补充依赖关系和技术说明
- 移除不再相关或已过期的条目

### 5. Backlog 精化会议（Backlog Refinement）

- 定期（建议每 Sprint 1-2 次）与团队一起评审 Backlog
- 确保 Sprint Backlog 有足够的"就绪"条目
- 调整优先级和估算

## 输入

- **需求来源**: 用户反馈、市场信息、内部需求、技术债
- **团队容量**: 团队的开发速度和可用资源
- **业务目标**: 产品战略和业务优先级
- **现有 Backlog**: 当前的产品待办列表

## 输出

- **整理后的 Backlog**: 按优先级排序的待办列表
- **优先级报告**: 优先级排序的依据和结果
- **就绪条目**: 已细化、可进入 Sprint 的条目
- **归档条目**: 已移除或延后的条目记录

## 注意事项

- **Backlog 是活文档**：它不是一成不变的，需要持续维护和调整。
- **保持适度大小**：Backlog 不应过长，超过 100 条时应考虑清理。
- **优先级动态调整**：市场和用户需求变化时，及时调整优先级。
- **让团队参与**：Backlog 精化是 PO 和团队的协作过程，不是 PO 一个人的事。
- **聚焦价值交付**：优先级排序的核心是最大化价值交付，而非"谁的声音大"。
- **记录决策原因**：调整优先级时记录原因，便于回溯和解释。
