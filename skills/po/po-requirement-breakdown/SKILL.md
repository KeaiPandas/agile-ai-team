# 需求拆解

## 概述

将产品需求（用户故事、功能需求、技术需求）拆解为 Agent 可直接执行的结构化 Task。每个 Task 必须包含明确的优先级（P0/P1/P2）、大小标签（S/M/L）、详细描述和可验证的验收标准。这是整个 AI Agent 开发流程的起点——PO 产出 Task，下游 Agent 才能按序领取和执行。

**在流程中的位置：** 产品愿景 → **需求拆解** → Backlog 管理

## 触发条件

- 收到新的产品需求或功能描述
- 现有需求发生变化需要重新拆解
- Scrum Master 反馈某个 Task 粒度不合理，需要细化或合并
- 里程碑规划前需要补充 backlog 内容

## 输入

- 产品需求文档（PRD）、用户故事、功能描述
- 技术架构说明（如有）
- 现有 `backlog/product-backlog.md`（避免重复创建）
- 约束条件（技术栈、性能要求、截止时间等）

## 流程

1. **理解需求全貌**：通读需求文档，明确核心目标和约束条件
2. **识别功能边界**：将需求拆分为独立、可交付的功能单元
3. **确定优先级**：
   - **P0（必须完成）**：核心功能，缺少则产品无法使用
   - **P1（重要）**：重要功能，显著影响用户体验
   - **P2（锦上添花）**：优化项，可延后交付
4. **评估大小标签**：
   - **S（小型）**：单文件修改、简单配置、文档更新，预计 <1h
   - **M（中型）**：涉及多文件、多模块交互，预计 1-4h
   - **L（大型）**：跨模块重构、核心功能实现，预计 >4h
5. **编写验收标准**：每个 Task 至少 2 条验收标准，必须是可测试、可验证的
6. **分配 Task ID**：按递增序号分配（T-001, T-002, ...），查看 backlog 避免冲突
7. **写入 Backlog**：将 Task 按优先级添加到 `backlog/product-backlog.md`
8. **输出摘要**：向请求方报告拆解结果，包含 Task 数量、总大小估算

## 输出

- 更新后的 `backlog/product-backlog.md`，包含新增 Task
- 拆解摘要（Task 数量、优先级分布、大小分布）

## 文件规范

### Task 拆解输出格式

每个 Task 在 backlog 中的格式：

```markdown
| T-XXX | Task 描述 | S/M/L | todo | - |
```

### 完整 Task 定义（单独文件时）

每个 Task 可选维护单独文件 `backlog/tasks/T-XXX.md`：

```markdown
# T-XXX: Task 标题

## 基本信息
- **优先级:** P0 / P1 / P2
- **大小:** S / M / L
- **状态:** todo / in-progress / review / done
- **负责人:** dev / qe / -
- **相关 Task:** T-XXX（如有依赖）

## 描述
详细说明需要做什么。

## 验收标准
1. [ ] 可验证的验收条件 1
2. [ ] 可验证的验收条件 2
3. [ ] 可验证的验收条件 3

## 备注
补充信息（技术约束、参考链接等）。
```

## 注意事项

- **粒度控制**：Task 不能太大（超过 L 应继续拆分），也不能太小（不足 S 应合并）
- **验收标准必须可测试**：避免模糊描述如"用户体验良好"，改为"页面加载时间 < 2s"
- **避免隐含依赖**：Task 之间的依赖关系必须显式标注
- **保持 backlog 干净**：过时的 Task 应标记废弃而非删除
- **大小评估参考代码量而非时间**：Agent 的执行速度与人不同，S/M/L 反映任务复杂度而非工时
