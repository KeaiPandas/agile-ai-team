# PO Agent - 需求拆解技能

## 概述

本技能指导 PO Agent 将高层次的业务需求、产品愿景或用户反馈转化为结构化、可执行的 User Story。需求拆解是连接"做什么"和"怎么做"的关键桥梁。好的拆解能让开发团队清晰理解需求、准确估算工作量、高效交付正确的产品功能。

## 触发条件

- 用户提出新的产品需求或功能愿景
- 需要将模糊的需求转化为具体的开发任务
- 收到"拆解需求"、"生成 User Story"、"分析需求"、"需求分解"等指令
- 产品规划阶段需要将 Epic 拆解为 Story
- Sprint Planning 前需要准备就绪的 User Story

## 流程

### 1. 需求收集与理解

- 与需求提出方深入沟通，理解业务背景和真实意图
- 明确目标用户和使用场景
- 识别核心痛点和期望价值
- 梳理相关的业务规则和约束条件

### 2. Epic 拆解

如果需求较大，先拆解为 Epic（大特性），再进一步拆为 Story：
- 按用户角色拆分（不同角色的不同需求）
- 按功能模块拆分（独立的业务能力）
- 按时间阶段拆分（MVP → 完整版）
- 确保每个 Epic 有独立可交付的价值

### 3. 编写 User Story

使用标准格式：
```
作为 <角色>，
我想要 <功能>，
以便于 <价值/目的>。
```

每个 Story 需满足 INVEST 原则：
- **I**ndependent: 独立可交付
- **N**egotiable: 可协商细节
- **V**aluable: 有明确价值
- **E**stimable: 可估算大小
- **S**mall: 足够小（1 个 Sprint 可完成）
- **T**estable: 可测试验证

### 4. 定义验收标准（Acceptance Criteria）

为每个 Story 编写明确的验收标准：
- 使用 Given-When-Then 格式描述
- 覆盖正常流程和异常场景
- 包含边界条件和约束
- 确保可测试、可验证

### 5. 补充上下文

- UI/UX 设计参考（如有）
- API 接口需求（如有）
- 数据模型变更（如有）
- 技术依赖和前置条件

## 输入

- **原始需求**: 用户反馈、业务需求、产品愿景
- **用户画像**: 目标用户角色和使用场景
- **业务规则**: 相关的业务约束和规则
- **技术约束**: 技术栈限制、性能要求等

## 输出

- **Epic 列表**: 高层次的需求分组
- **User Story 集合**: 规范化的用户故事（含标题、描述、角色）
- **验收标准**: 每个 Story 的验收条件
- **依赖关系**: Story 之间的依赖和执行顺序建议

## 注意事项

- **需求拆解不是一次性的**：随着理解的深入，可能需要多次迭代细化。
- **与开发团队保持沟通**：拆解过程中听取开发团队的技术建议和可行性评估。
- **避免过度拆解**：Story 太小会增加管理开销，太大则难以估算和交付。
- **聚焦用户价值**：每个 Story 都应回答"这给用户带来什么价值"。
- **验收标准要具体**：模糊的验收标准会导致开发偏差和验收争议。
- **记录假设**：明确标注需求中的假设条件，便于后续验证。
