# Release Agent - 发布日志生成技能

## 概述

本技能指导 Release Agent 自动生成清晰、规范、有价值的发布日志。发布日志是版本发布的重要交付物，它帮助用户了解新版本的变更内容、帮助团队成员了解版本演进历史。一份好的发布日志应该对齐变更来源（Issue/PR），按类别组织，语言面向用户而非开发者。

## 触发条件

- 版本发布完成后需要生成发布日志
- 需要整理版本的变更内容
- 收到"发布日志"、"changelog"、"release notes"、"更新日志"等指令
- Sprint 结束需要总结本周期交付内容
- 需要为外部用户准备版本说明

## 流程

### 1. 收集变更信息

- 从 Git 历史中提取 commit 记录
- 从 PR/MR 中获取变更描述和关联 Issue
- 收集已合并 PR 的分类信息
- 整理 Bug 修复记录和功能变更记录

### 2. 变更分类

按类型分类整理：
- ✨ **新功能 (Features)**: 新增的功能和特性
- 🐛 **Bug 修复 (Bug Fixes)**: 修复的缺陷
- 🔧 **改进 (Improvements)**: 功能改进和优化
- 💥 **破坏性变更 (Breaking Changes)**: 不兼容的变更
- 🗑️ **移除 (Removed)**: 移除的功能或 API
- 📖 **文档 (Documentation)**: 文档更新

### 3. 评估影响级别

- **Major**: 破坏性变更、重大功能更新
- **Minor**: 新功能、重要改进
- **Patch**: Bug 修复、小改进

### 4. 编写发布日志

标准格式（推荐 Keep a Changelog 格式）：
```markdown
# [版本号] - YYYY-MM-DD

## ✨ 新功能
- 功能描述 (关联 #Issue编号)

## 🐛 Bug 修复
- 修复描述 (关联 #PR编号)

## 🔧 改进
- 改进描述

## 💥 破坏性变更
- 变更描述及迁移指南

## 📋 升级说明
- 升级注意事项和步骤
```

### 5. 审查和发布

- 请 PO 和团队负责人审查发布日志
- 确认外部发布的内容没有敏感信息
- 发布到 GitHub Release、文档站等渠道
- 通知相关利益方

## 输入

- **Git 历史**: 版本间的 commit 和 merge 记录
- **PR/MR**: 已合并的 Pull Request 列表
- **Issue 列表**: 已关闭的 Issue 记录
- **版本号**: 发布的版本号和日期

## 输出

- **Changelog**: Markdown 格式的发布日志
- **Release Notes**: 面向用户的版本说明
- **版本标签**: Git tag 和版本发布信息

## 注意事项

- **面向用户而非开发者**：发布日志的语言应该让最终用户理解，而非只写给开发看。
- **关联原始来源**：每个变更都应关联到对应的 Issue 或 PR，便于追溯。
- **标注破坏性变更**：破坏性变更必须醒目标注，并提供迁移指南。
- **保持一致性**：使用统一的格式和分类标准，便于用户阅读和比较。
- **及时更新**：每个版本发布后都应生成对应的日志，不要积压。
- **Changelog 也是文档**：它是产品的变更历史，是团队和用户的重要参考资料。
