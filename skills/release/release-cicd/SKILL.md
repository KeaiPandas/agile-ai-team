# CI/CD 管理

## 概述

Release Agent 管理项目的持续集成和持续交付流程，包括代码构建、自动化测试、环境部署和发布管理。CI/CD 是代码从开发到上线的自动化通道，确保每次发布都是可预测、可重复的。

**在流程中的位置：** 发布日志 → **CI/CD 部署** → 生产环境

## 触发条件

- 代码合并到 main 分支（触发 CI）
- 里程碑完成，需要发布新版本（触发 CD）
- 环境配置变更
- 紧急修复需要快速部署

## 输入

- 构建配置文件（如 `Dockerfile`、`vite.config.ts` 等）
- 部署配置（环境变量、服务器地址等）
- `CHANGELOG.md`（版本信息）
- 环境管理文件

## 流程

### CI 流程（持续集成）
1. **代码检查**：Lint、类型检查、格式检查
2. **运行测试**：单元测试、集成测试
3. **构建**：编译打包生成可部署产物
4. **生成报告**：测试报告、覆盖率报告、构建日志
5. **结果通知**：CI 通过/失败通知

### CD 流程（持续交付）
1. **准备发布**：
   - 确认 CI 全部通过
   - 确认 Changelog 已更新
   - 确认版本号正确
2. **构建发布产物**：
   - 生产环境配置构建
   - 生成 Docker 镜像（如适用）
   - 打 tag 并推送
3. **部署到目标环境**：
   - 开发环境：每次合并自动部署
   - 预发布环境：里程碑完成后部署，进行验收
   - 生产环境：验收通过后手动触发部署
4. **部署验证**：
   - 健康检查（接口可达、关键功能正常）
   - 冒烟测试
5. **发布确认**：更新发布记录

## 输出

- 构建产物
- 部署状态
- 发布记录（`release/release-log.md`）
- 环境状态

## 文件规范

### 发布记录格式

路径：`release/release-log.md`

```markdown
# 发布记录

## v0.2.0
- **发布时间:** 2026-03-29 16:00
- **发布者:** Release Agent
- **环境:** Production
- **状态:** 成功 / 失败 / 回滚

### 发布内容
- 对应里程碑: M-002 用户认证
- 变更 Task: T-001, T-002, T-003
- 修复 Bug: BUG-001

### 部署信息
- **构建号:** build-42
- **Docker 镜像:** registry.example.com/app:v0.2.0
- **部署耗时:** 2m 15s

### 验证结果
- ✅ 健康检查通过
- ✅ 冒烟测试通过

### 回滚记录（如有）
- **回滚时间:** YYYY-MM-DD HH:MM
- **回滚版本:** v0.1.0
- **回滚原因:** 部署后发现问题
```

### 环境管理

路径：`release/environments.md`

```markdown
# 环境配置

## 开发环境 (dev)
- 地址: http://localhost:3000
- 自动部署: 每次 push 到 main

## 预发布环境 (staging)
- 地址: https://staging.example.com
- 自动部署: 里程碑 Tag 推送

## 生产环境 (production)
- 地址: https://example.com
- 部署方式: 手动触发
```

## 注意事项

- **CI 失败必须阻塞合并**：不允许 CI 失败的代码进入主分支
- **环境隔离**：开发、预发布、生产环境严格隔离
- **部署可回滚**：每次部署前确保可以快速回滚到上一版本
- **发布前确认**：生产环境部署必须有确认步骤
- **记录每次发布**：无论成功失败都记录，便于问题追溯
