# QE Agent - Bug 报告技能

## 概述

本技能指导 QE Agent 规范地记录和跟踪软件缺陷。一份高质量的 Bug 报告能让开发者快速理解问题、准确定位原因、高效修复缺陷。QE Agent 应以"让开发者一看就懂、一做就复现"为目标来编写 Bug 报告。

## 触发条件

- 测试过程中发现 Bug
- 需要提交缺陷报告
- 收到"bug"、"缺陷"、"提交 bug"、"bug report"等指令
- 用户反馈的异常行为需要记录
- 自动化测试失败需要分析和记录

## 流程

### 1. Bug 确认与分类

- **确认是 Bug**：区分 Bug、需求变更和建议改进
- **复现确认**：至少能稳定复现 2 次以上
- **排除环境因素**：确认不是环境配置、网络或数据问题
- **查重**：搜索已有的 Bug 报告，避免重复提交

### 2. Bug 分级

按严重程度分类：
- **P0 - 致命**: 系统崩溃、数据丢失、核心功能不可用
- **P1 - 严重**: 主要功能异常、有明显影响的问题
- **P2 - 一般**: 功能有偏差但可绕过、UI 显示问题
- **P3 - 轻微**: 拼写错误、样式微调、优化建议

### 3. 编写 Bug 报告

标准 Bug 报告格式：

```
**标题**: [简明扼要描述 Bug]

**环境**:
- 浏览器/系统/版本
- 测试环境（staging/production）
- 复现频率（必现/偶发/概率）

**复现步骤**:
1. 步骤 1
2. 步骤 2
3. 步骤 3

**预期结果**: 描述正确的行为应该是什么

**实际结果**: 描述当前的实际行为

**截图/录屏**: 附上相关的截图或录屏

**附加信息**:
- 错误日志
- 控制台输出
- 相关 API 响应
```

### 4. 提交与跟踪

- 提交到 Bug 跟踪系统（Jira、GitHub Issues 等）
- 设置正确的优先级和标签
- 指定负责人（如有）
- 跟踪 Bug 状态变化

### 5. 验证关闭

- 开发者修复后进行验证
- 确认 Bug 已修复且未引入新问题
- 在回归测试中验证相关功能
- 关闭 Bug 报告

## 输入

- **Bug 现象**: 观察到的异常行为
- **复现步骤**: 复现问题的操作序列
- **环境信息**: 浏览器、系统、版本等
- **相关日志**: 错误日志、控制台输出等

## 输出

- **Bug 报告**: 规范化的缺陷报告
- **Bug 状态跟踪**: 从发现到关闭的完整跟踪记录
- **验证结果**: 修复后的验证结论

## 注意事项

- **复现步骤是核心**：好的复现步骤是 Bug 报告的灵魂，必须精确到每个操作。
- **不要主观臆断原因**：描述现象和复现方式，让开发者判断原因。
- **附带足够信息**：截图、录屏、日志——信息越多，定位越快。
- **一个 Bug 一个报告**：不要在一个报告中塞多个不相关的问题。
- **标题要有信息量**：不要写"有 Bug"，要写"登录页面输入特殊字符后页面白屏"。
- **及时更新状态**：验证结果及时反馈，不要让 Bug 停留在"已修复"状态无人确认。
